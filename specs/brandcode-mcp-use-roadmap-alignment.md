# Brandcode MCP Use Roadmap Alignment

**Status:** Current alignment note for M001 v0.1 proof
**Date:** 2026-05-08
**Depends on:** `specs/brandcode-mcp-phase-0-lock.md`, UCS portable runtime package artifact work through S037, M001-L01 hosted smoke harness

## Purpose

Align the hosted **Use** MCP roadmap with the implementation that now exists in this repo and the portable runtime work landing in UCS. The Phase 0 lock remains valid: `@brandsystem/mcp` authors the runtime, while `@brandcode/mcp` serves a governed hosted brand over Streamable HTTP through exactly eight tools.

This is not a new MCP surface. It is a contract and proof update so Brandcode MCP consumes the runtime substrate UCS actually has now: live Brand package JSON, Full Brand Runtime package artifacts, private-custody redaction, AgentRun history, and append-only hosted review input.

## Product Object Spine

Brandcode MCP must keep these objects distinct:

- **Official Brand:** the canonical governed brand in Brandcode Studio / Runtime Admin. Hosted MCP can read from it, but cannot mutate it.
- **Full Brand Runtime:** the default v0.1 Use MCP object. It is the complete governed runtime package surfaced from UCS and read by `brand_runtime`, `brand_search`, `brand_check`, status, asset, and history flows.
- **Production-Approved Asset:** an asset approved for production use. Asset tools may expose package-safe metadata and delivery references for these assets; they must not surface raw private provider URLs.
- **selected Brand Kit:** a narrowed kit selected for a specific context. It is not the default hosted MCP object until UCS has durable hosted selected-kit publish/share URL truth.
- **campaign/exploratory kit:** a temporary or exploratory collection. It must not masquerade as the Official Brand or Full Brand Runtime in v0.1.
- **hosted patch/review request:** append-only operator input, such as `brand_feedback` observation/proposal records. It can request review or propose a change, but it is not canonical governance mutation.
- **`DESIGN.md`:** an adapter/readable brief generated by the Build MCP. It is useful for humans and design-agent adapters, but runtime authority lives in the structured `.brand` / hosted runtime artifacts.

## Current State

### In This Repo

- `src/hosted/`, `api/[slug].ts`, and `bin/brandcode-mcp.mjs` provide the hosted Brandcode MCP scaffold.
- The hosted server registers exactly the locked eight-tool Phase 0 surface in the Phase 0 order:
  1. `brand_runtime`
  2. `brand_search`
  3. `brand_check`
  4. `brand_status`
  5. `list_brand_assets`
  6. `get_brand_asset`
  7. `brand_feedback`
  8. `brand_history`
- `brand_status` reports all eight hosted tools as implemented, with `remaining_stubs: []`.
- Per-tool scope enforcement is executable: `read` gates runtime/search/status/assets/history, `check` gates `brand_check`, and `feedback` gates `brand_feedback`.
- `brand_runtime` and `brand_status` read the hosted Brand package through `/api/brand/hosted/{slug}/pull`.
- `brand_runtime` recognizes the flat UCS `brandInstance` package shape and normalizes a minimal runtime-like object when no top-level compiled runtime exists.
- `brand_search` queries hosted package knowledge from narratives, proof points, application rules, brand phrases, readiness, and capabilities with provenance/status fields.
- `list_brand_assets` and `get_brand_asset` expose package-safe asset metadata, official/runtime/production-approved posture, and redacted custody-safe delivery refs.
- `brand_check` validates text, color, font, and CSS against hosted package governance and returns a read-only verdict/finding shape.
- `brand_feedback` builds an append-only UCS AgentRun history entry for observation/proposal review input. It does not approve, apply, or mutate canon.
- `brand_history` reads UCS AgentRun history with provider `mcp`, surface `mcp-hosted`, limit/cursor inputs, and compact receipt-chain summaries.
- General hosted telemetry POST remains deferred. Explicit `brand_feedback` append is the only current write-like hosted behavior, and it writes review input only.
- M001-L01 added `npm run smoke:hosted-mcp` as the repo-native repeatable proof command for hosted MCP initialize/list/tools/core calls and insufficient-scope behavior.

### In UCS

UCS has moved beyond the original April scaffold assumptions:

- The service-token pull route can return a compiled Brand package with `runtime` derived from `brandInstance`.
- Full Brand Runtime package artifacts use a brand-scoped namespace:
  - `{brand}/runtime/packages/{brand}-brand-runtime-{hash}.zip`
  - `{brand}/runtime/packages/{brand}-brand-runtime-{hash}.receipt.json`
  - `{brand}/runtime/packages/latest.json`
- Brandcode and C5 have current downloadable Full Brand Runtime artifacts with private-custody redaction and external-consumer proof.
- Runtime Admin distinguishes Official Brand, Production-Approved Asset, selected Brand Kit, campaign/exploratory kit, and Full Brand Runtime. Hosted MCP responses and docs must preserve those distinctions.
- Selected Brand Kit work is progressing as a local artifact / CLI / target-path contract first. Hosted share URL and configured publish are downstream, not v0.1 assumptions.
- AgentRun history exists at `/api/brand/hosted/{slug}/agent/history`; POST expects an `AgentRunHistoryEntry` under `entry`.

## Alignment Principles

1. **Use MCP serves the live Full Brand Runtime by default.** `brand_runtime`, `brand_search`, `brand_check`, and asset catalog reads start from the service-token Brand package JSON. The hosted ZIP is the external-consumer delivery artifact, not the primary MCP read path.
2. **The eight-tool lock stays closed.** Build, extraction, compile, local filesystem, selected-kit publish, and canonical mutation tools remain outside `@brandcode/mcp` v0.1.
3. **Selected/campaign kits are not the v0.1 default.** Do not make selected Brand Kits or campaign/exploratory kits appear as the default hosted brand until UCS ships hosted selected-kit publish/share URL truth.
4. **Package artifact truth belongs in status and assets.** `brand_status` reports whether Full Brand Runtime artifact metadata exists. Asset tools expose package-safe delivery refs and custody posture, not raw private provider URLs.
5. **Checks are read-only governance evaluation.** `brand_check` can judge draft text/color/font/CSS against hosted governance, but cannot mutate runtime state or create official approval.
6. **Feedback is append-only hosted review input.** `brand_feedback` records observations/proposals as review or hosted patch-request input. Runtime Admin / Brand Console remains the place where canonical changes are approved.
7. **History is scoped and receipt-aware.** `brand_history` reads AgentRun history scoped by API key/brand permissions and preserves compact receipt-chain context where UCS provides it.
8. **`DESIGN.md` is an adapter brief.** Build MCP may generate it for human/design-agent readability, but hosted Use MCP authority comes from the structured runtime/package contract.

## Remaining Proof Gates

### M001-L03: Staging Route And Client Access

Goal: prove `https://mcp.staging.brandcode.studio/{slug}` directly from a real MCP client, not only through Vercel preview or local component tests.

Still blocked by:

- DNS/alias/certificate setup for `mcp.staging.brandcode.studio`.
- Vercel deployment-protection or bypass-token posture that allows normal MCP clients to reach the route.
- Env-provided live smoke inputs: `BRANDCODE_MCP_SMOKE_URL`, `BRANDCODE_MCP_SMOKE_FULL_KEY`, and preferably `BRANDCODE_MCP_SMOKE_READ_KEY`.

Proof command:

```bash
BRANDCODE_MCP_SMOKE_URL="https://mcp.staging.brandcode.studio/{slug}" \
BRANDCODE_MCP_SMOKE_FULL_KEY="bck_test_..." \
BRANDCODE_MCP_SMOKE_READ_KEY="bck_test_..." \
npm run smoke:hosted-mcp
```

Required proof:

- MCP `initialize`
- `tools/list`
- exact locked eight-tool order
- `brand_status`
- `brand_runtime`
- `brand_search`
- `list_brand_assets`
- optional `get_brand_asset` when `BRANDCODE_MCP_SMOKE_ASSET_ID` is available
- `brand_check`
- `brand_history`
- `brand_feedback`, unless intentionally skipped with `BRANDCODE_MCP_SMOKE_SKIP_FEEDBACK=1`
- insufficient-scope behavior for `brand_check` and `brand_feedback` with a read-only key

Capture hosted route proof separately from human-visible UI proof. Runtime Admin UI proof is useful, but it cannot substitute for MCP client route proof.

### M001-L04: Feedback Append Proof

Goal: prove `brand_feedback` writes an append-only UCS history entry with review posture and without canonical governance mutation.

Still blocked by:

- A real UCS service token configured for the hosted MCP deployment.
- A reachable hosted MCP endpoint and scoped feedback key.

Acceptance:

- `brand_feedback` returns `append_status: recorded`.
- The stored record is an observation/proposal review input, not an Official Brand mutation.
- The proof preserves key receipt facts without exposing raw private provider URLs.

### M001-L05: v0.1 Release Readiness

Goal: cut release/readiness posture only after source, hosted, scope, history, and feedback claims are all true.

Release gate:

- All eight locked tools remain implemented and no ninth tool is introduced.
- Auth failures, slug authorization, per-tool scopes, upstream errors, not-found states, and rate-limit posture are covered by tests or explicit documented deferrals.
- Staging proves a real MCP client can call the live hosted route.
- `brand_feedback` append is proved or explicitly deferred from v0.1 with a named blocker.
- Hosted responses distinguish local component/API proof, hosted route proof, and human-visible UI proof.
- Package metadata and directory listings cross-link `@brandsystem/mcp` for Build and `@brandcode/mcp` for Use without migration pressure.

## Explicit Deferrals

- Build/extract/compile tools stay in `@brandsystem/mcp`.
- Canonical governance mutation stays in Brand Console / Runtime Admin.
- selected Brand Kit hosted publish/share is not part of v0.1 unless UCS ships durable hosted selected-kit artifact truth first.
- campaign/exploratory kits do not become the default hosted Brandcode MCP runtime object.
- Public unauthenticated reads remain rejected for v0.1.
- Separate `brand_proof_check` and `brand_narrative_routing` remain folded into `brand_check` and `brand_search`.
- General hosted AgentRun telemetry remains deferred unless it can emit the valid UCS `{ entry: AgentRunHistoryEntry }` contract without blocking tool responses.
